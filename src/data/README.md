# AI Gateway model metadata

`ai-gateway-models.yaml` is the source of truth for every model Neon's AI Gateway supports. It feeds the docs page, `models.json`, and the filterable models page: none of those should hand-maintain their own model list or specs, they all render from this file.

## Why this file exists

The previous setup (`models.json` generated from models.dev) had drifted badly from what Neon's gateway actually serves: stale entries for removed models, missing entries for models that had shipped, and no way to catch either. This file is kept honest against Neon's own model catalog, its internal configuration for each model, and live requests to the real gateway whenever those disagree or don't cover something. Everything in this file traces back to a real source. Nothing is a guess.

## Why live checks, not just internal config

Internal configuration says what a model is supposed to do. It doesn't always match what the gateway actually does on a live request, and it doesn't cover everything a customer needs to know. So every entry gets checked, or re-checked, against the real gateway, not just against internal records. Live checks are what catch:

- **Whether a model is actually reachable.** Internal lifecycle flags can lag behind reality in both directions: a model marked fine can be dead, and a model marked deprecated can still work.
- **Real limits, not marketing figures.** Context windows and max output tokens are confirmed by requesting past the stated limit and reading the real cap back from the error, not by trusting a rounded number from a spec sheet.
- **Whether a claimed capability actually works.** Tool use, reasoning, web search, and multimodal input only get listed if a live call to the model confirms it, not because a config file says it should.
- **The exact shape of provider-specific parameters**, like which reasoning effort levels a model actually accepts.

When a live result and internal config disagree, the live result wins.

## What's in here

Only what a customer needs to pick and use a model: capabilities, specs, pricing (where known), lifecycle status in plain language, and runnable code samples per SDK. No internal implementation or infrastructure details.

## Schema, briefly

Two-level map: `provider → modelId → fields`. Per model: `displayName`, `openWeights`, `description`, `recommendedFor`/`alternatives`, `contextWindow`/`maxOutputTokens`, `modalities`, `capabilities`, `reasoning` (if applicable, the actual parameter shape, not just "yes it reasons"), `pricing` (real numbers for proprietary models where published; `free_preview`/`tbd` otherwise, open-weight models never get a fabricated price), `releaseDate`, `status`/`statusNote`, `dialects`, `codeSamples`, and `lastVerified`.

## How code samples are generated

`codeSamples` holds one block per SDK (AI SDK, Mastra, Python, TypeScript, cURL) for every model. These aren't written by hand: they're generated by running the same snippet-generation logic the Neon Console uses to build its own AI Gateway code samples, so what's here always matches what a customer would see in the product itself, just pointed at a placeholder endpoint instead of a real one.

## Keeping it current

Run `/update-ai-models`. It diffs the model list for new/changed/removed models, runs a lifecycle health-check against a live probe for everything else, and never auto-commits: it hands you a diff to review.
