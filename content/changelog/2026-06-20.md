## Neon backend for apps and agents is in private preview

Storage, Functions, and AI Gateway are now in private preview. Early users are already building with them at no charge.

- **Storage**: S3-compatible object storage that branches with your database. Every branch gets its own isolated storage, so your test and preview environments stay fully self-contained. Available at no charge during private preview.
- **Functions**: Serverless functions that run alongside Postgres. No separate deployment, no cold-start penalty from spinning up a distant service.
- **AI Gateway**: A single proxy for frontier models from Anthropic, OpenAI, Google, Meta, and more, built into your Neon project. During private preview, all models are available at no charge. That includes `claude-sonnet-4-6`, `gpt-5`, `gemini-2-5-flash`, and the full catalog listed in [AI Gateway models](/docs/ai-gateway/models).

## neon.ts: infrastructure as code for your Neon project

`neon.ts` is a TypeScript config file you commit to your repo. It declares which Neon services your project uses and defines a branch policy function that runs at branch creation time to configure TTL, compute size, and scale-to-zero settings for each new branch.

```ts
import { defineConfig } from "@neondatabase/config/v1";

export default defineConfig({
  auth: true,
  dataApi: true,
  branch: (branch) => {
    if (branch.isDefault) return {};
    if (!branch.exists) return { ttl: "7d" };
    return {};
  },
});
```

Apply it with `neonctl deploy`, preview changes first with `neonctl config plan`. The `@neondatabase/env` package gives you type-safe access to the environment variables each declared service injects, so missing or misconfigured variables surface at build time rather than runtime.

See the [neon.ts reference](/docs/reference/neon-ts) for the full config schema, or read the [blog post](https://neon.com/blog/introducing-neon-ts) for a full walkthrough.

## New guide: export Neon metrics to SigNoz

SigNoz contributed a guide showing how to export your Postgres logs and metrics to [SigNoz](https://signoz.io) using the Neon OpenTelemetry integration. Once set up, your database telemetry lands alongside your application data in SigNoz, making it easier to spot and investigate performance issues across your stack.
