## Neon backend for apps and agents is now in beta

Neon Storage, Functions, and AI Gateway are graduating from private preview to beta. No signup or waitlist: everyone can start building a complete backend on new or existing projects in AWS US East (Ohio) today.

The point isn't three separate services. You declare your whole backend in one `neon.ts` file, and it branches with your data: fork a branch and you get an isolated copy of your database, files, functions, and gateway.

## New TypeScript SDK for the Neon API

We're introducing [`@neon/sdk`](https://www.npmjs.com/package/@neon/sdk) 1.1, a new official TypeScript SDK for the **Neon Platform API**. The Platform API is the REST API for managing projects, branches, databases, and the new backend services (Storage, Functions, and AI Gateway) in code, with the same capabilities as the [Neon Console](https://console.neon.tech). It replaces [`@neondatabase/api-client`](https://www.npmjs.com/package/@neondatabase/api-client) as the recommended client. The legacy package continues to work for existing apps. `@neon/sdk` is OpenAPI-generated, uses native `fetch` with zero runtime dependencies, and requires Node.js 20.19+.

Call `createNeonClient({ apiKey })` for namespaced methods (`neon.projects`, `neon.branches`, `neon.storage`, `neon.functions`, `neon.aiGateway`, and more), typed `{ data, error }` results, and workflow helpers such as `createAndConnect`. Mutations return background operations; helpers poll until provisioning finishes, and any method accepts `{ waitForReadiness: true }` to block until ready. A `raw` layer exposes every Platform API endpoint.

```bash
npm install @neon/sdk
```

```typescript
import { createNeonClient } from "@neon/sdk";

const neon = createNeonClient({ apiKey: process.env.NEON_API_KEY! });

// Workflow helper: create, poll until ready, return a connection string
const { data, error } = await neon.projects.createAndConnect({ name: "my-app" });
if (error) throw error;
const { project, connectionString } = data;

// Or wait on any mutation with waitForReadiness
const { data: branch, error: branchError } = await neon.branches.create(
  project.id,
  { name: "preview" },
  { waitForReadiness: true }
);
if (branchError) throw branchError;

// The new backend services are namespaced too
await neon.storage.buckets.create(project.id, branch.id, { name: "uploads" });
```

See the [TypeScript SDK documentation](/docs/reference/typescript-sdk) and [migration guide](/docs/reference/migrate-api-client-to-sdk) for setup, API reference, and moving from `@neondatabase/api-client`.

## Passkey support

You can now sign in to Neon with a passkey instead of a 2FA code. Add a passkey from **Account settings** and use your device's built-in biometrics, like Touch ID or Windows Hello, or a security key to verify it's you. Passkeys satisfy organization-level 2FA requirements, so admins can let members enroll in either 2FA or a passkey to comply. See [Manage your Neon account](/docs/manage/accounts#passkeys) for setup steps.

## New `neon diff` CLI command

We've added a `diff` command to the Neon CLI, letting you (and your agents) quickly see schema changes between your current branch and another branch. It fits into a branch-first development workflow alongside `neon link`, `neon checkout`, `neon status`, and `neon deploy`.

1. `neon link` — link to a Neon project
2. `neon checkout dev-1` — create/checkout a dev branch
3. Do the dev work
4. `neon diff main` — sanity check the schema changes made against main

```diff
--- a/neondb
+++ b/neondb
@@ -12,6 +12,11 @@

 CREATE TABLE public.orders (
     id integer NOT NULL,
     customer_id integer NOT NULL,
-    status text NOT NULL
+    status text NOT NULL,
+    discount_code character varying(20)
 );

+CREATE INDEX orders_discount_code_idx ON public.orders USING btree (discount_code);
```

Tell your agents about `neon diff`.

## Choose your production branch when connecting Vercel

When you connect a Neon project to the [Neon-Managed Vercel Integration](/docs/guides/neon-managed-vercel-integration), you can now select which Neon branch supplies credentials for Vercel's **Production** environment (`DATABASE_URL`, `DATABASE_URL_UNPOOLED`, and related variables). Use this when your live data lives on a branch other than the project default.

Preview branches and the optional `vercel-dev` branch still use your project's **default branch** as their parent. [Learn more](/docs/guides/neon-managed-vercel-integration#choose-a-production-branch).

## New NAT gateway IPs and VPC endpoint services in US East (Ohio), Europe (London), and Asia Pacific (Singapore)

We've expanded infrastructure capacity in the AWS US East (Ohio) (`us-east-2`), Europe (London) (`eu-west-2`), and Asia Pacific (Singapore) (`ap-southeast-1`) regions with new NAT gateway IP addresses and new VPC endpoint service addresses for Private Networking.

<Admonition type="tip" title="Update your IP allowlists">
If you have IP allowlists on external systems that Neon connects to, **update those allowlists to include the new NAT gateway addresses**. Connections may be affected intermittently if traffic routes through non-allowlisted NAT gateways.
</Admonition>

If you use Private Networking in these regions, you can now use the additional VPC endpoint service addresses for enhanced capacity and reliability. See the [Regions documentation](/docs/introduction/regions#aws-nat-gateway-ip-addresses) for the complete list of NAT gateway IPs and the [Private Networking guide](/docs/guides/neon-private-networking) for VPC endpoint service addresses by region.
