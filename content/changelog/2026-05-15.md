## Neon Auth: new plugins and settings

Neon is building out the full backend stack for Postgres-powered applications, and authentication is central to that. Three new Neon Auth updates this week give you more flexibility in how users sign in and how your app presents itself:

**Magic link sign-in:** Neon Auth now supports the Magic link plugin for passwordless sign-in. Users receive an email with a link to authenticate without a password. If you're using Neon Auth UI components, enable it with a single prop:

```tsx
<NeonAuthUIProvider
  authClient={authClient}
  magicLink // [!code ++]
>
```

Enable the plugin from **Auth** > **Plugins** in the Neon Console or via the API. See [Magic link plugin](/docs/auth/guides/plugins/magic-link) for setup details.

**Phone number sign-in:** The new Phone Number plugin lets existing users sign in with a one-time password (OTP) delivered by SMS. You bring your own SMS provider and connect it via a `send.otp` webhook. The sign-in flow is a two-step SDK call:

```ts
// Step 1: send the OTP
await authClient.phoneNumber.sendOtp({ phoneNumber });

// Step 2: verify the code and sign in
await authClient.phoneNumber.verify({ phoneNumber, code });
```

Enable the plugin from **Auth** > **Plugins** in the Console or via the API. See [Phone number plugin](/docs/auth/guides/plugins/phone-number) for setup details.

**Custom application name:** You can now set a custom application name that appears in user-facing auth messages like verification emails and password resets. Configure it per branch under **Auth** > **Configuration** in the Console, so development and production branches can use different names. See [Application name](/docs/auth/production-checklist#application-name) for details.

If you're preparing a Neon Auth integration for production, the [Auth production checklist](/docs/auth/production-checklist) covers everything you need: trusted domains, email provider setup, OAuth credentials, and more.
</Admonition>

## Neon Auth MCP tools

Two new tools are now available in the [Neon MCP server](/docs/ai/neon-mcp-server) for managing Neon Auth:

- **`configure_neon_auth`:** Configure your Neon Auth setup, including OAuth providers, email providers, and authentication methods.
- **`get_neon_auth_config`:** Retrieve your current Neon Auth configuration, including integration metadata.

If you haven't set up the Neon MCP server yet, one command gets you started:

```bash
npx neonctl@latest init
```

Once connected, you can set up authentication for your app using natural language in your AI editor. For example:

```
Set up Neon Auth for my project. Enable Google OAuth and email/password sign-in,
and set the application name to "My App".
```

Your AI assistant will call the Neon MCP tools to configure everything without leaving your editor. See [Neon MCP server](/docs/ai/neon-mcp-server) for full setup details.

## Neon and Stripe Projects

Earlier this year, Neon announced support for [Stripe Projects](https://neon.com/blog/neon-works-with-stripe-projects-for-agentic-provisioning), Stripe's platform for AI-native development. With the Stripe Projects CLI installed, your AI agent can provision a Neon Postgres database in a single command:

```bash
stripe projects add neon/postgres
```

Stripe Projects handles authentication and writes the credentials directly to your `.env` file. No dashboards, no copy-pasting connection strings. From there, you can prompt your agent to build with the database immediately:

```text
Build a Next.js app that stores user trips in the Neon database.
Use the DATABASE_URL from the environment.
```

The new [Neon with Stripe Projects guide](/guides/projects-dev-neon) walks through a complete example, from provisioning to deployment.
