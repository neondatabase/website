## Simplified Lakebase Search setup

Installing the [Lakebase Search](/docs/ai/lakebase-search) extensions just got simpler on Postgres 16 and later. Lakebase Search brings hybrid vector and full-text search into the same Postgres database, and you can now install both extensions, `lakebase_vector` and `lakebase_text`, by running the `CREATE EXTENSION` statements directly:

```sql
CREATE EXTENSION IF NOT EXISTS lakebase_vector CASCADE;
CREATE EXTENSION IF NOT EXISTS lakebase_text CASCADE;
```

Previously, you had to load the shared preload libraries with a Neon API call before you could install the extensions. That step is no longer required. See the [Lakebase Search overview](/docs/ai/lakebase-search) to get started.

_P.S. Curious how it works? Our engineering deep dive covers why the usual `pgvector` + GIN setup breaks down at scale and how `lakebase_ann` (IVF + RaBitQ) and `lakebase_bm25` (Block-Max WAND) are designed for Neon's separated compute and storage, so indexes live on object storage and stay ready across scale-to-zero and branching. Read [Lakebase Search: vector and BM25 on Neon](/blog/lakebase-search-on-neon)._

## Neon CLI: new name and two new commands

**A shorter name.** You can now install the Neon CLI from npm as `neon` instead of `neonctl`: `npm i -g neon` (or run it with `npx neon@latest`). All commands are now documented as `neon` rather than `neonctl`. If you already use `neonctl`, nothing changes: it's the same CLI, `neonctl` still works as a command, and no migration or re-authentication is needed. See the [Neon CLI install guide](/docs/cli/install).

**`neon config init`.** Scaffolds a starter [`neon.ts`](/docs/reference/neon-ts) policy file and installs the `@neondatabase/config` and `@neondatabase/env` packages, so you can start managing your branches declaratively in one step. It runs entirely locally, and `neon link` now offers to run it as its final step.

```bash
neon config init
```

**`neon status`.** A top-level alias for `neon config status`. Add `--current-branch` to print just the branch pinned in your local `.neon` file, with no network request, login, or analytics. It's fast and quiet enough to drive a shell prompt:

```bash
neon status --current-branch
```

See the [config command reference](/docs/cli/config#current-branch) for a ready-to-paste [starship](https://starship.rs) prompt snippet that shows your current Neon branch.

## Use Neon Storage with the Files SDK

[Neon Storage](/docs/storage/overview) now has a first-class adapter in the [Files SDK](https://files-sdk.dev), the open-source library that gives you one upload, download, and presigned-URL API across S3, R2, GCS, and more. The `neon` adapter is wired up from the `AWS_*` variables Neon injects, so pointing your storage backend at Neon is a one-line config change. See the [with-files-sdk example](https://github.com/neondatabase/examples/tree/main/with-files-sdk) for a minimal script that uploads files to a branch-scoped bucket.

## Community: Neon Testing now supports Bun Test

[Neon Testing](https://www.npmjs.com/package/neon-testing), the community-built integration testing library by [Mikael Lirbank](https://www.lirbank.com/), shipped v3.0.0 with support for [Bun Test](https://bun.sh/docs/cli/test) alongside Vitest. Each test runs against its own isolated Neon branch, with `DATABASE_URL` set up and torn down automatically, so your tests hit the same schema and constraints as production without mocks or a shared local database.

```bash
npm install --save-dev neon-testing
```

- View the package on npm: [neon-testing](https://www.npmjs.com/package/neon-testing)
- Read the release notes: [v3.0.0 on GitHub](https://github.com/starmode-base/neon-testing/releases)

<details>
<summary>**Neon API**</summary>

- You can now update a snapshot's expiration with the [Update snapshot](https://api-docs.neon.tech/reference/updatesnapshot) endpoint. Set `expires_at` to a future timestamp to change the retention deadline, or send `null` to remove it so the snapshot never expires. Previously, expiration could only be set when the snapshot was created.

  ```bash
  curl --request PATCH \
    --url 'https://console.neon.tech/api/v2/projects/{project_id}/snapshots/{snapshot_id}' \
    --header 'authorization: Bearer $NEON_API_KEY' \
    --header 'content-type: application/json' \
    --data '{
      "snapshot": {
        "expires_at": "2026-12-31T00:00:00Z"
      }
    }'
  ```
