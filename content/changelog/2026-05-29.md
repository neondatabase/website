## Postgres 18 for newly created Neon projects

![Create PG18 project](/docs/changelog/create_project_18.png)

Postgres 18 is now the default for newly created Neon projects. Neon continues to support Postgres 14, 15, 16, and 17, if you prefer to stick with those. For Neon's Postgres version support policy, see [Postgres Version Support](/docs/postgresql/postgres-version-policy).

## Replayable AI agents with Neon snapshots

AI agents that mutate databases create a difficult failure mode: when something goes wrong, you can't just retry the prompt. The data has already changed.

Neon snapshots solve this. By pairing a snapshot of your database with a serialized copy of your agent's execution state, you create a checkpoint: a durable record of both what the database looked like and what the agent was planning to do at that exact moment. If the agent takes a wrong turn, restore the snapshot and replay the run from the checkpoint. Because Neon preserves your connection string after a restore, no app restarts or reconfiguration are needed.

The pattern unlocks a range of capabilities:

- **Interrupt before destructive calls:** Pause the agent at a sensitive tool call, take a snapshot, then approve or reject. If rejected, the database rolls back and the agent tries a different path.
- **Time-travel debugging:** Restore any historical snapshot to an isolated branch and replay a week-old run without touching production data.
- **Auditable execution:** Link every agent trace to a snapshot ID for full reproducibility.

The new guide walks through a complete implementation using the OpenAI Agents SDK and Neon's Snapshots API: [Build replayable AI agents with Neon snapshots](/guides/replayable-ai-agents).

Building agents at scale? The [Neon Agent Plan](/docs/introduction/agent-plan) is designed for platforms that provision and manage databases programmatically, from per-agent-run isolation to high-volume fleet management.
